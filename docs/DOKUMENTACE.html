1. Přehled aplikace

1.1 O aplikaci

Operating Room Management System je komplexní webová aplikace pro efektivní řízení operačních sálů v zdravotnických zařízeních. Aplikace umožňuje reálné sledování stavu jednotlivých sálů, správu pracovních časů, automatické notifikace operačního personálu a detailní analýzu efektivity provozu. Všechny statistiky a údaje vycházejí z reálných dat v databázi, nikoliv z předpokládaných či mockovaných hodnot.

1.2 Hlavní cíle

  • Centralizované řízení operačních sálů a jejich statusu
  • Reálný monitoring stavu jednotlivých sálů v reálném čase
  • Automatické notifikace při kritických stavech (pozdní příchod, nepřipravený pacient)
  • Detailní statistiky a reporting využití sálů
  • Řízení workflow procesu operačních postupů
  • Správa notifikací a komunikace se zdravotnickým personálem

1.3 Klíčové vlastnosti

  • Reálné monitorování: Live dashboard s aktuálním stavem všech sálů, animované přechody stavů
  • Workflow management: Detailné sledování fází operačních postupů (příprava, chirurgický výkon, sešívání, údržba)
  • Notifikační systém: Automatické notifikace pro pozdní příjezdy, nepřipravené pacienty, ostatní problémy
  • Statistiky z reálných dat: Analýza vytížení sálů, efektivity, časových řad, vše z tabulky room_status_history
  • Správa konfigurace: Flexibilní nastavení pracovních hodin, workflow kroků, kritérií notifikací
  • Historické údaje: Úplná historie všech stavů a aktivit v databázi s RLS ochranou

Realtime updates

  1. Supabase realtime subscription naslouchá změnám v tabulce operating_rooms

  3. React state se aktualizuje a UI se překreslí s animacemi

  1. Uživatel vybere časový úsek (den/týden/měsíc/rok)

5. Dashboard - Hlavní modul

5.1 Přehled

Dashboard (App.tsx) je hlavní komponentou aplikace. Slouží jako centrální orchestrator, který:

  • Spravuje globální state (rooms, settings, workflowSteps)
  • Naslouchá realtime změnám v Supabase
  • Přepíná mezi jednotlivými moduly
  • Řídí autentizaci a autorizaci
  • Handleuje error stavy a loading

5.2 Struktura kódu

export const Dashboard: React.FC = () => {
  const [activeModule, setActiveModule] = useState('rooms');
  const [rooms, setRooms] = useState([]);
  const [workflowSteps, setWorkflowSteps] = useState([]);
  
  // Načtení dat z databáze
  useEffect(() => {
    loadInitialData();
  }, []);
  
  // Realtime subscription
  useEffect(() => {
    const subscription = setupRealtimeUpdates(setRooms);
    return () => subscription?.unsubscribe();
  }, []);
  
  // Render podle aktivního modulu
  return (
    <div>
      {activeModule === 'rooms' && <OperatingRoomsManager />}
      {activeModule === 'stats' && <StatisticsModule />}
      {activeModule === 'settings' && <SettingsPage />}
    </div>
  );
};

5.3 Realtime Subscriptions

Dashboard se přihlašuje k realtime aktualizacím:

const setupRealtimeUpdates = (setRooms: (rooms: OperatingRoom[]) => void) => {
  return supabase
    .channel('public:operating_rooms')
    .on('postgres_changes', 
      { event: '*', schema: 'public', table: 'operating_rooms' },
      (payload) => {
        // Aktualizace lokálního stavu
        setRooms(prev => updateRoomInList(prev, payload.new));
      }
    )
    .subscribe();
};

8. StatisticsModule - Statistiky a reporting

8.1 Důležité: Pouze reálná data!

Všechny statistiky v tomto modulu se počítají VÝHRADNĚ z reálných dat uložených v databázi (tabulka room_status_history). Nepoužívají se žádná mockovaná, statická ani předpokládaná data.

8.2 Sekce "Sály"

  • Operace za 24h: Počet operací na jednotlivý sál za poslední 24 hodin
  • Využití (%): Procento času, kdy sál aktivně provádí operaci
  • Fronta: Počet pacientů čekajících na operaci
  • Scatterplot: Graf vztahu mezi počtem operací a vyu žitím na sál

8.3 Sekce "Fáze"

  • Průměrné trvání fází: Z reálných dat (v minutách). Počítá se z tabulky room_status_history kde duration_seconds se převede na minuty
  • Procentuální zastoupení: Jak velkou část času zaberá každá fáze (Příprava, Chirurgický výkon, Sešívání, Údržba)
  • Kumulativní graf: Příspěvek jednotlivých fází k celkovému času operace
  • Heatmapa fází: Procentuální zastoupení fází na jednotlivý sál

8.4 Sekce "Heatmapa"

  • Heatmapa den × hodina: Vizualizace vytížení v jednotlivých hodinách a dnech (0-24h)
  • Denní průměr: Průměrné vytížení dle dne v týdnu (pondělí-neděle)
  • Hodinový průběh: Vytížení v jednotlivých hodinách (pracovní vs. víkend)
  • Statistiky: Špička, minimum, průměr, pracovní hodiny (respektují weekly_schedule)

8.5 Filtrace dat podle období

Období Rozsah Seskupení Pracovní doby

8.7 Počítání pracovní doby

Funkce dayMinutes(room) počítá pracovní minuty daného dne v týdnu z weekly_schedule:

// Pro UPS sály (emergency): 24h × 60 = 1440 minut
// Pro normální sály: (endHour - startHour) × 60 - odpočet startMinute, endMinute

// Příklad:
// Pondělí: 7:00 - 19:00 = 12 hodin = 720 minut
// Sobota: disabled = 0 minut
// Neděle (UPS): 0:00 - 24:00 = 1440 minut

11. Pracovní procesy a stavy

11.1 Workflow stavy operačního sálu

Každý sál prochází definovanými fázemi během operace:

idle (Volný) ↓ preparing (Příprava sálu a pacienta) ↓ procedure (Chirurgický výkon) ↓ closing (Sešívání a epilog) ↓ maintenance (Čištění a sterilizace) ↓ idle (Volný - připraven na další operaci)

11.2 Detailní popis fází

Idle (Volný)

  • Sál není v provozu
  • Je připraven na další operaci
  • Barva: Šedá

Preparing (Příprava)

  • Doba: 15-30 minut (configurable)
  • Aktivity: Příprava pacienta, sterilizace, anestézie
  • Odpovědnost: Zdravotní sestra, Anestezioloд
  • Barva: Modrá

Procedure (Chirurgický výkon)

  • Doba: Variabilní dle typu operace (30-300 minut)
  • Aktivity: Samotný chirurgický zákrok
  • Odpovědnost: Chirurg + tým
  • Barva: Zelená
  • Důležité: Tato fáze se počítá jako "Utilization" v statistikách

Closing (Sešívání)

  • Doba: 10-20 minut
  • Aktivity: Zavření ran, příprava na převoz
  • Odpovědnost: Chirurg, Zdravotní sestra
  • Barva: Oranžová

Maintenance (Údržba)

  • Doba: 15-45 minut
  • Aktivity: Úklid, dezinfekce, sterilizace nástrojů
  • Odpovědnost: Údržbový personál
  • Barva: Červená

11.3 Notifikační stavy

Trigger: Pozdní příchod operatéra

  • Podmínka: Operátor se nedostavil v čas (nastavitelný offset)
  • Akce: Email notifikace vedoucího oddělení
  • Záznam: notifications_log s kategoriím "late_surgeon"

Trigger: Nepřipravený pacient

  • Podmínka: Pacient není připraven v čase
  • Akce: Email notifikace zdravotní sestry
  • Záznam: notifications_log s kategoriím "patient_not_ready"

Trigger: Pozdní příjezd z oddělení

  • Podmínka: Přesun pacienta z oddělení zpožďuje operaci
  • Akce: Email notifikace koordinátora
  • Záznam: notifications_log s kategoriím "late_arrival"

13. Řešení problémů

13.1 Realtime updates se nezobrazují

Problém: Změny stavu sálů se neaktualizují v reálném čase.
  • Ověřit, že realtime je aktivní v Supabase dashboardu
  • Zkontrolovat console.log pro chyby připojení
  • Ověřit síťové připojení a CORS nastavení
  • Restartovat dev server

13.2 Notifikace se neposílají

Problém: Email notifikace se neposílají příjemcům.
  • Ověřit RESEND_API_KEY v environment proměnných
  • Zkontrolovat, zda nejsou překročeny limity Resend API
  • Ověřit, že email adresy jsou validní
  • Podívat se na error logy v Vercel dashboardu

13.3 Statistiky se nezměnily při filtrování

Problém: Statistiky se zobrazují stejně bez ohledu na zvolené období.
  • Zkontrolovat, že se data zapisují do room_status_history
  • Ověřit datumové filtry - jsou data v daném rozsahu?
  • Zkontrolovat weekly_schedule - je nastaven správně?
  • Zkontrolovat dev tools → Network → fetchStatusHistory API call

13.4 TypeError: Cannot read property weekly_schedule

Problém: Chyba při přístupu k weekly_schedule.
  • Ověřit, že operační sál má nastavenou weekly_schedule
  • Zkontrolovat, že DEFAULT_WEEKLY_SCHEDULE je importován
  • Ověřit, že struktura obsahuje všechny 7 dní

13.5 Performance problémy - pomalé statistiky

Problém: Načítání statistik trvá dlouho.
  • Ověřit indexy na room_id a timestamp v room_status_history
  • Zvážit archivaci starších dat (>1 roku)
  • Zvýšit limit dotazu nebo implementovat pagination

13.6 Neoprávněný přístup k datům

Problém: Chyba autentizace nebo "permission denied".
  • Ověřit RLS politiky v Supabase
  • Zkontrolovat, že token je validní
  • Ověřit role uživatele a oprávnění
  • Restartovat autentizaci

13.7 Debug nástroje

  • Supabase Studio: SQL editor a data browser
  • Browser DevTools: Console, Network, Application tabs
  • React DevTools: Component profiler a state inspection
  • Vercel Analytics: Performance monitoring a error tracking
