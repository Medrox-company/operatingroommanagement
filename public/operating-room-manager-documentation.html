01 — Přehled

Co aplikace dělá

OperatingRoom Manager je real-time systém pro řízení operačních sálů ve zdravotnickém zařízení. Sleduje workflow operací krok za krokem, koordinuje personál, generuje statistiky a notifikace pro management. Změny se synchronizují přes Supabase Realtime na všechna zařízení do 2 sekund.

Real-time stav

Stav sálů

Aktuální fáze workflow každého sálu (8 hlavních fází + 4 speciální stavy), informace o pacientovi, personálu, výkonu a odhadovaný čas dokončení.

Plánování

Timeline a směny

Plán operací po hodinách na týdenním pohledu, rozvrh směn personálu a obsazení sálů s pravidelnými/speciálními pracovními hodinami.

Analytika

Statistiky

Vytíženost sálů, počet operací podle oddělení, průměrné délky fází, nedělané operace, exporty dat. Filtrace podle data, oddělení, sálu.

Personál

Správa zaměstnanců

Lékaři, sestry, anesteziologové, jejich dovednostní úrovně (L1–L3, A, SR, N, S), externí spolupráce, dostupnost, dovolené, neschopenky.

Notifikace

Automatické alerty

Email upozornění managementu na pozdní příjezd, urgentní operace, denní report, nedostupnost personálu nebo pacienta.

Konfigurace

Admin režim

Kompletní konfigurace oddělení, sálů, statusů workflow, modulů přístupu pro role, vzhledu a integrace e-mailů.

Klíčové funkce

  • 8-stupňový workflow operace: Sál připraven → Příjezd na sál → Začátek anestezie → Chirurgický výkon → Ukončení výkonu → Ukončení anestezie → Odjezd ze sálu → Úklid sálu.
  • 4 speciální stavy: Pauza, Hygienický režim, Volání pacienta, Příjezd do operačního traktu.
  • Real-time synchronizace přes Supabase Realtime — změna stavu se objeví na všech klientech do 2 sekund.
  • Optimistické UI — UI se mění okamžitě, server potvrdí v pozadí, debounce 2 s zabraňuje flickeringu z duplicitních realtime událostí.
  • Audio alarm pro urgentní operace, automaticky zastaví po deaktivaci.
  • Mobilní aplikace přes responsive layout (MobileShell) s vyhrazenými pohledy pro tablet/mobil.
  • Role-based access — 6 rolí (admin, primar, management, cos, aro, user), každá s vlastním seznamem povolených modulů.
  • Per-module řízení — administrátor může povolit/zakázat moduly nebo přístup rolí přímo z UI.
  • Drag & drop řazení sálů (perzistuje v operating_rooms.sort_order).
  • Týdenní rozvrhy sálů s konfigurovatelnými hodinami pro každý den.
  • PDF/HTML export dokumentace pomocí html2canvas + jsPDF.
  • Customizovatelné pozadí — solid/linear/radial gradient, image overlay, blur, opacity (uložené v DB).

Datový tok

  1. Client mount → AuthProvider → fetch /api/auth/me (HttpOnly session cookie) → uživatel + dostupné moduly z app_modules.
  2. Initial fetch → fetch('/api/rooms') → server volá fetchOperatingRooms() z lib/db.ts → Postgres přes Supabase JS klient → JSON.
  3. Realtime subscription → subscribeToOperatingRooms(...) v App.tsx → naslouchá UPDATE/INSERT/DELETE na operating_rooms, schedules, staff.
  4. UI mutace → optimistická úprava lokálního state → updateOperatingRoom(id, changes) → DB UPDATE → realtime echo se ignoruje díky recentLocalUpdates (debounce 2 s).
  5. Background settings → app_settings tabulka (1 řádek) → custom event backgroundSettingsChanged propaguje napříč moduly bez fetch round-tripu.
03 — Autentizace

Login flow a role

Hesla jsou hashovaná v Postgres pomocí pgcrypto bcrypt. Server-side ověření přes Supabase RPC verify_user_password. Session je vlastní podepsaný JWT token v HTTP-only Secure cookie (žádný localStorage).

    API-->>L: { user }

Bezpečnostní opatření

  • bcrypt hash hesel s pgcrypto, ne otevřená v DB.
  • HttpOnly Secure cookie — token nelze přečíst z JavaScriptu, není uložen v localStorage.
  • SameSite=Lax — ochrana proti CSRF u většiny scénářů.
  • JWT s HMAC-SHA256, podepsaný SUPABASE_JWT_SECRET; serverová validace v lib/auth/session.ts.
  • Rate limit 10 pokusů / 5 min / IP v lib/auth/rate-limit.ts.
  • Validace vstupu — délka e-mailu ≤ 255, hesla ≤ 512, regex pro e-mail formát.
  • Row Level Security aktivní na všech 14 aplikačních tabulkách.

Speciální stavy

Pauza
Operace pozastavena (např. čekání na výsledek vyšetření). Tlačítko v RoomDetail.
Hygienický režim
Zvýšený hygienický standard po septické operaci.
Volání pacienta
Pacient byl povolán na sál — uloží patient_called_at.
Příjezd do operačního traktu
Pacient dorazil do operačního traktu — patient_arrived_at.

Příznaky stavů sálu

  • is_emergency — urgentní operace, spustí audio alarm na všech klientech přes useEmergencyAlert hook.
  • is_septic — septická operace, vyžaduje speciální úklid + hygienický režim.
  • is_locked — sál uzamčen (např. údržba), nelze plánovat operace.
  • is_paused — operace pozastavena, čas se nezapočítává do statistik fáze.

Audit historie

Každá změna fáze je zaznamenána do operating_rooms.status_history (jsonb pole) a paralelně do room_status_history tabulky pro statistiky. Ukončené operace se ukládají do completed_operations jsonb pole pro denní přehled.

06 — API

API Reference

Všechny endpointy běží na Next.js runtime nodejs, vyžadují platnou session cookie (kromě /api/auth/login) a vrací application/json.

Auth

POST/api/auth/login

Body: { email: string, password: string }

Odpověď 200: { user: {...} } + cookie session=...

Chyby: 400 (bad input), 401 (špatné údaje), 429 (rate limit), 503 (DB).

POST/api/auth/logout

Smaže session cookie. Odpověď: { success: true }.

GET/api/auth/me

Pokud platná session, vrací { user }; jinak 401.

Operační sály

GET/api/rooms

Vrátí pole OperatingRoom[] seřazené podle sort_order, fallback name. Network resilience: 3 retry na transient error.

POST/api/operating-rooms/reorder

Body: { rooms: [{ id, sort_order }] } — atomicky aktualizuje pořadí všech sálů.

Workflow statusy

GET/api/workflow-statuses

Konfigurace všech stavů (8 hlavních + 4 speciální).

POST/api/workflow-statuses

Vytvoří nový status. Pouze admin role.

PUT/api/workflow-statuses

Aktualizuje stav podle id.

DELETE/api/workflow-statuses?id=…

Smaže status (kromě defaultních systémových).

Management kontakty & Notifikace

GET/api/management-contacts · POST · PUT · DELETE

CRUD nad management_contacts včetně granularních notify_* příznaků.

POST/api/send-notification

Body: { type, roomId, customReason? }. Odešle e-mail vybraným kontaktům s notify_<type> = true. Log do notifications_log.

Admin

GET / PUT/api/admin/facility

Údaje o zařízení (název, IČO, adresa, kontakty).

GET/api/admin/export-data

Exportuje všechna data (sály, personál, oddělení, statusy) jako JSON soubor.

POST/api/admin/import-data

Naimportuje JSON dump zpět do databáze (validace, transakce).

POST/api/admin/reset-data

Resetuje runtime data (zachová konfiguraci) — pro demo / testovací prostředí.

10 — Návod

Jak používat aplikaci

Krok-za-krokem návody pro každou roli a typický workflow den na operačním sále. Pro detailní informace o jednotlivých fázích viz sekce 05 Workflow.

A. První přihlášení

  1. Otevři aplikaci v prohlížeči (Chrome / Edge / Safari, JS povolen).
  2. Na úvodní obrazovce klikni „Přihlášení do systému“.
  3. Zadej e-mail a heslo (viz sekce 03 — Demo účty), nebo použij rychlé tlačítko jedné z 6 rolí.
  4. Po úspěšném přihlášení se otevře Dashboard s aktuálním stavem všech sálů.
  5. V mobilním zobrazení (≤ 768 px) je layout zjednodušený s bottom navigation a optimalizovaný pro dotyk.

B. Standardní workflow operace (pohled COS / sálová sestra)

  1. Klikni na kartu sálu v Dashboardu — otevře se RoomDetail.
  2. Tlačítko „Volání pacienta“ — záznam patient_called_at, čas se objeví v hlavičce.
  3. „Příjezd do operačního traktu“ — pacient přijel, patient_arrived_at.
  4. „Příjezd na sál“ — posun fáze 0 → 1, začíná operation_started_at.
  5. Postupně odklikávej tlačítka pro fáze 2–7 (anestezie → operace → ukončení → odjezd → úklid).
  6. Pro urgentní operaci aktivuj přepínač is_emergency — spustí audio alarm na všech klientech.
  7. Pro septickou operaci přepni is_septic — automaticky zařadí hygienický režim po skončení.
  8. Pauza — speciální stav, čas se nezapočítává do statistik aktuální fáze.

C. Plánování (pohled management / primariát)

  1. V Sidebar klikni na Timeline.
  2. Zobrazí se Gantt-like plán dne / týdne s sály jako řádky a hodinami jako sloupce.
  3. Filtruj podle oddělení (TRA, CHIR, …) v hlavičce.
  4. Klikni na blok pro náhled detailu, drag pro úpravu času (pokud má role oprávnění).

D. Statistiky (pohled management)

  1. Sidebar → Statistiky.
  2. Vyber časové období (Den / Týden / Měsíc / Rok / Custom).
  3. KPI karty: počet operací, průměrná délka, vytíženost sálů, septické / urgentní / pauzy.
  4. Graf po hodinách / dnech, breakdown podle oddělení a sálu.
  5. Tlačítko Export CSV stáhne data pro Excel / další zpracování.

E. Notifikace (pohled admin / management)

  1. Admin → Management kontakty — přidej e-maily managementu.
  2. Pro každý kontakt nastav granularně, které notifikace chce dostávat (notify_late_arrival, notify_emergencies, notify_daily_reports, …).
  3. Notifikace odešle POST /api/send-notification automaticky podle workflow událostí.
  4. Historie odeslaných notifikací v tabulce notifications_log.
  5. Pro povolení e-mailů nastav RESEND_API_KEY ENV, jinak se notifikace pouze logují.

F. Administrace (pouze role admin)

  1. Operační sály — přidat / přejmenovat / smazat sály, drag & drop pořadí.
  2. Oddělení — definice TRA / CHIR / NEUROCHIR…, sub-oddělení (HPB, cévní, dětské, mammo, prokto).
  3. Workflow statusy — barva, ikona, default trvání, viditelnost ve Timeline / Detail / Statistikách.
  4. Plány směn — kdo na jakém sále kdy.
  5. Systémová nastavení — povolení modulů per-role (např. zakázat ARO Statistics), údaje o zařízení, export / import / reset dat.

G. Klávesové zkratky (desktop)

  • Esc — zavřít otevřený dialog / RoomDetail / picker.
  • ⌘ / Ctrl + K — quick navigation (pokud aktivní).
  • ⌘ / Ctrl + P — tisk aktuálního pohledu.
11 — Bezpečnost

Bezpečnostní opatření

Aplikace zpracovává citlivá zdravotnická data — autentizace, hesla, RLS, rate limiting a defense-in-depth jsou nastavené na všech vrstvách.

Hesla

Hashování bcrypt

Postgres extension pgcrypto s bcrypt cost 10. Heslo nikdy neopustí DB v plain textu — RPC verify_user_password dělá compare server-side. Aplikační kód nikdy nečte password_hash.

Session

JWT v HttpOnly cookie

HMAC-SHA256 podepsaný JWT se ukládá do HttpOnly Secure SameSite=Lax cookie. Žádný localStorage. Ochrana proti XSS odcizení tokenu.

Rate limit

10 / 5 min / IP

lib/auth/rate-limit.ts in-memory token bucket per IP. Po překročení vrací 429 s Retry-After hlavičkou.

RLS

Row Level Security

Na všech 14 aplikačních tabulkách je RLS aktivní. Klient anon key vidí pouze data povolená politikami; serverové mutace přes service_role bypassují RLS, ale jsou validovány v API vrstvě.

Vstupy

Validace

Email regex, max-length kontroly (255 / 512), trim + lower-case před comparison. Vrácené chybové hlášky neodhalují informace o existenci účtu.

Transport

HTTPS only

Vercel deployment automaticky vynucuje HTTPS, HSTS hlavičky, Secure cookie flag. Žádné mixed-content scénáře.

Doporučené produkční nastavení

  • Po prvním deployu změň výchozí hesla všech demo účtů (admin123 apod.).
  • Zúž RLS politiky podle organizační struktury, pokud aplikaci používá více oddělení.
  • Aktivuj Supabase audit log v Dashboardu pro sledování DB operací.
  • Nastav Vercel Web Application Firewall (WAF) pro pokročilou ochranu.
  • Pravidelný backup Postgres (Supabase v Pro plánu má daily PITR).
  • Vyžaduj 2FA pro admin účty (Supabase managed nebo vlastní implementace).
12 — Troubleshooting

Časté problémy & řešení

Problém

Nelze se přihlásit (401)

Příčina: špatné heslo, neaktivní účet, nebo špatný formát e-mailu.

Řešení: ověř velká/malá písmena, zkontroluj app_users.is_active = true, ověř SUPABASE_JWT_SECRET ENV.

Problém

429 Too Many Requests

Příčina: 10 neúspěšných pokusů o přihlášení během 5 minut z téže IP.

Řešení: počkej 5 minut, nebo restartuj server (in-memory bucket se resetuje).

Problém

Realtime updates se neaktualizují

Příčina: nepovolený Supabase Realtime na tabulce, vypršený WebSocket.

Řešení: Supabase Dashboard → Database → Replication → zaškrtni operating_rooms a schedules; refresh stránky obnoví připojení.

Problém

Pořadí sálů se po refreshi resetuje

Příčina: chyběl mapping sort_order v transformRoom nebo .order('name') místo 'sort_order'.

Řešení: opraveno v aktuální verzi lib/db.ts; ověř, že operating_rooms.sort_order má hodnoty (ne NULL).

Problém

E-mail notifikace se neodesílají

Příčina: chybí RESEND_API_KEY, nebo příjemce nemá notify_<type> = true.

Řešení: nastav ENV; v Admin → Management kontakty zaškrtni požadované notifikace.

Problém

Bílá obrazovka po deployi

Příčina: chybí NEXT_PUBLIC_SUPABASE_URL nebo NEXT_PUBLIC_SUPABASE_ANON_KEY.

Řešení: Vercel → Project → Settings → Environment Variables — přidej a redeploy.

Problém

Audio alarm nehraje na iPhone

Příčina: Safari blokuje autoplay audio bez user gesture.

Řešení: klepni kdekoli v aplikaci po načtení (jednorázové gesture stačí pro celou session).

Problém

Pomalé načítání statistik

Příčina: velký room_status_history bez indexů.

Řešení: přidej index CREATE INDEX ON room_status_history (operating_room_id, timestamp); archivuj řádky starší 12 měsíců.

Diagnostika v prohlížeči

  • DevTools → Network — zkontroluj /api/auth/me, /api/rooms 200 OK.
  • DevTools → Application → Cookies — ověř existenci cookie session s HttpOnly flagem.
  • Console — chyby s prefixem [v0] nebo [Auth] indikují fail-soft scénáře.
  • Supabase Logs — Dashboard → Logs → API / Auth / Database.
